# ADR-007: Deterministic registry convergence via forced synchronous import

Status: Accepted (Option Y; implemented in PR #50)
Date: 2026-07-20 (two-tier design)
Revised: 2026-07-21 (Option Y, after the live import spike)

This ADR changed its decision. It was first proposed (2026-07-20) as a two-tier
split -- assert gateway/backend synchronously, monitor registry convergence
asynchronously -- because forcing convergence looked unachievable. A live spike
on 2026-07-21 proved otherwise, so the decision is now Option Y: force
convergence synchronously and assert it in the gate. The superseded two-tier
design is retained under Alternatives, with the evidence that overturned it.

## Context

Scenario S3 exposes the MCP server through Azure API Center. The v1 tracer's live
gate (`.github/workflows/ephemeral-env.yml` + `scripts/gate/invoke-and-assert.ps1`)
runs apply -> call -> destroy against an ephemeral environment. Step `[5]`
originally asserted, with a 300 s bounded poll on the data-plane
`/workspaces/default/v0.1/servers` endpoint, that the server appeared, and failed
the run otherwise. That was doubly wrong, established live and against Learn:

1. **The data-plane `/v0.1/servers` surface is not a headless-bearer API.** On
   the first live gate (2026-07-20) both an anonymous GET and a valid
   Data-Reader-role `azure-apicenter.net` bearer token returned 401. Verified
   2026-07-21: `/v0.1/servers` is not in the published data-plane REST reference;
   its auth is the portal access mode (anonymous or interactive Entra SPA
   sign-in), and the Data Reader role's documented purpose is authorizing
   interactive portal sign-in. A 401 there is expected, not a wiring bug.

2. **Auto-sync is unbounded relative to the gate.** API Center is populated the
   production-correct way, by one-way APIM -> API Center auto-sync, documented at
   "typically minutes, but up to 24 hours." An ephemeral gate cannot wait that
   out, so asserting convergence on auto-sync alone is a flaky required check --
   one slow-sync day turns the gate red for reasons unrelated to the diff and
   trains the team to ignore red.

3. **The MCP server DOES converge into the control-plane inventory, as kind=mcp.**
   Live (2026-07-21, run 29803582476): the `2024-06-01-preview` control-plane
   `apis` list returns the synced server as `{title: <server_name>, kind: "mcp",
   name: <auto-generated>}`. So `kind=mcp` exists in the live API despite being
   absent from every documented enum (the live API is ahead of its docs). This
   control-plane surface is readable with the `management.azure.com` token the
   gate already holds -- the correct read surface, unlike `/v0.1/servers`.

4. **The pivot: forced synchronous import works.** The 2026-07-21 spike ran
   `az apic import-from-apim` (apic-extension 1.1.0) against the live env. It is a
   blocking LRO (~34 s, exit 0), it is idempotent, and it COEXISTS with the active
   auto-sync `apiSources` link with no error and no duplicate entry. This is the
   mechanism the original two-tier design assumed did not exist.

## Decision

Make registry convergence deterministic (Option Y): force it synchronously, then
assert it. Retire the two-tier / async-monitor plan.

- **Force convergence before the call stage.** A workflow step
  ("Force registry convergence (import-from-apim, Option Y)") runs
  `az apic import-from-apim` for the MCP server API after the s2 apply. This is a
  synchronous, idempotent LRO that lands the server in the API Center inventory,
  and coexists with the production auto-sync link (verified live). Auto-sync stays
  the production headline; the import is a CI-only determinism mechanism layered
  on top, not a replacement.

- **Assert convergence in the gate.** Gate step `[5]` reads the control-plane
  `apis` inventory (`management.azure.com`, `2024-06-01-preview`) and ASSERTS an
  entry with `title == ServerName && kind == "mcp"` (the entry's own name is
  auto-generated). FATAL if absent, with a short bounded retry as a safety margin
  for residual projection lag -- not an eventual-consistency wait. It also probes
  `/v0.1/servers` anonymously to record the secure-by-default 401 posture, and
  captures the raw inventory as evidence.

- **No Tier 2.** Because convergence is forced and asserted synchronously, there
  is no eventual-consistency tail to monitor; the async nightly monitor is not
  built.

- **No guessed code ships.** The import command shape and its behavior (synchronous,
  idempotent, coexists, kind=mcp) are pinned from the installed extension and a
  live run, not inferred.

## Consequences

- Registry convergence is proven on every gate run, deterministically. A broken
  sync/import path fails the gate loudly instead of silently degrading.
- The gate gains a dependency on a PREVIEW CLI command in flux. apic-extension
  stable (1.1.0) exposes `import-from-apim`; the 1.2.0b3 beta renames it to
  `import apim` (with unverified params). Accepted risk: when the rename ships
  stable the import step breaks cleanly and visibly, which is preferable to
  guessing the renamed command; recorded as a re-check trigger in COMPATIBILITY.md.
- The gate spends ~34 s forcing the import and installs apic-extension on the
  runner. Acceptable for a live-gate that already runs for many minutes.
- The import is redundant with the production auto-sync path (both land the same
  entry). This is deliberate: auto-sync is what production relies on; the import
  only exists to make the ephemeral gate deterministic, and its idempotent
  coexistence with auto-sync (verified) is why layering it is safe.

## Alternatives considered

- **Two-tier: non-blocking gate evidence + async registry monitor (SUPERSEDED).**
  This ADR's original decision (2026-07-20). Registry step non-blocking; a nightly
  async monitor would catch slow convergence; Tier 2 designed but unimplemented on
  cost. Superseded 2026-07-21: the import spike proved forced synchronous
  convergence works and coexists with the link, so determinism is achievable in
  the gate itself -- strictly better than a monitored eventual-consistency gap
  that left registry regressions uncaught between nightly runs. The two-tier
  design was correct given what was known before the spike; the spike changed what
  was known.
- **Synchronous assertion on auto-sync alone (no forced import).** Rejected: the
  worst case is 24 h; a finite poll is a coin flip on sync latency -> flaky.
- **Assert the data-plane `/v0.1/servers` read.** Rejected: that surface 401s a
  headless bearer token (portal-auth-only); it is only probed anonymously for the
  secure-by-default posture.
- **Explicit azapi registration.** Rejected: no azapi/ARM MCP-registration payload
  exists and the `apis kind` enum omits `mcp`; a plain `apis` resource would apply
  green but never carry MCP semantics.
- **Portal-oracle capture / manual portal registration into a long-lived API
  Center.** Not needed now that `import-from-apim` is proven; kept as history.
